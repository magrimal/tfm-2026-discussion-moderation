El repositorio es un monorepo. El backend en Python contiene los agentes, el grafo, la API y las herramientas de evaluación; el directorio \texttt{dashboard} contiene el frontend en React; y los archivos de infraestructura describen los entornos de Idril y AWS. Los artefactos de cada ejecución se guardan en el sistema de archivos o en S3. Incluyen el manifiesto, los resultados por modelo e hilo y los mensajes necesarios para inspeccionar la ejecución.

\section{Pipeline de análisis y facilitación}
\label{sec:pipeline}

El modelo de intervención requiere tomar decisiones en secuencia y conservar los resultados de cada paso. A diferencia de una única llamada al modelo, este diseño permite inspeccionar cada decisión y que el agente de rol consulte el repertorio de técnicas antes de generar el texto.

La arquitectura es un pipeline multiagente implementado como un grafo dirigido, donde cada fase es un nodo con entradas y salidas definidas que se comunican a través de un estado compartido. Esta separación entre un orquestador y agentes especializados permite evaluar cada fase de forma independiente, ajustar el comportamiento por rol de facilitación sin afectar la clasificación, y auditar la lógica de cada fase por separado; sigue el patrón de los sistemas de tutoría multiagente \parencite{LippertEtAl2020}, donde el silencio ante el progreso de la discusión es el resultado esperado en la mayoría de las ejecuciones, no un caso límite.

La figura~\ref{fig:pipeline} muestra el grafo que ejecuta la implementación. Sus cuatro nodos son:

\begin{itemize}
    \item \texttt{Classification} caracteriza el hilo y le asigna un estado de la tabla~\ref{tab:estados}.
    \item \texttt{Intervention} decide si la evidencia del hilo justifica intervenir. Si la respuesta es negativa, la ejecución termina.
    \item \texttt{Orchestrator} selecciona el rol de facilitación cuando se ha decidido intervenir.
    \item \texttt{Role} elige una técnica y genera el texto. Las comprobaciones opcionales de la respuesta se ejecutan dentro de este nodo; si detectan un problema y quedan intentos, el control vuelve al orquestador.
\end{itemize}

Los experimentos no mantienen memoria persistente entre ejecuciones ni escriben historial de intervenciones. Cada caso se evalúa desde su instantánea del hilo. El código contiene una herramienta opcional para consultar historial durante otros modos de ejecución, pero esa capacidad no forma parte de la evidencia experimental presentada.

El sistema se implementó en Python 3.12 con pydantic-ai\footnote{\url{https://ai.pydantic.dev/}}. Se eligió porque reúne salidas estructuradas, validación con Pydantic, proveedores de LLM distintos y grafos con estado tipado. \texttt{pydantic\_graph} permite representar la topología de la figura~\ref{fig:pipeline}; cuando una respuesta no satisface el esquema declarado, la validación falla durante la ejecución y puede alimentar un reintento controlado. Esta combinación resultó útil para ejecutar el mismo pipeline con distintos proveedores.

La capa REST usa FastAPI y reutiliza los modelos Pydantic como contrato de la API. El panel es una aplicación React y la comunicación HTTP con fuentes externas usa httpx.

La observabilidad tiene dos vistas. El resultado conserva los mensajes de cada nodo para mostrarlos en el panel, mientras que Logfire registra duración, llamadas a herramientas y errores de validación como trazas OpenTelemetry. La figura~\ref{fig:logfire-development} muestra la jerarquía de una ejecución completa y la figura~\ref{fig:logfire-model-call} amplía una llamada al modelo para inspeccionar su entrada y su salida estructurada. La instrumentación es opcional y el pipeline funciona sin credenciales de Logfire.

El pipeline separa las dependencias de ejecución del estado de cada caso. Las dependencias agrupan configuración, clientes de modelos, acceso a la fuente y persistencia; se preparan antes de iniciar el grafo. El estado acumula el hilo, la clasificación, la decisión, el rol y la respuesta mientras avanza la ejecución. Esta separación permite probar los nodos con servicios controlados y evita mezclar configuración con resultados. Los puntos de extensión que sí usa el prototipo son los proveedores de LLM, las fuentes de hilos y el almacén de artefactos de experimentos.

Para desarrollo local, pruebas automatizadas y experimentos, los hilos se cargan desde escenarios definidos en el repositorio. Esto hace reproducibles las entradas y evita depender de una instancia externa durante la comparación de modelos.

Los agentes devuelven salidas estructuradas para que cada decisión quede en un campo inspeccionable. Como los modelos locales y externos no ofrecen el mismo soporte de herramientas, cada perfil indica si el esquema se transmite como herramienta o como instrucción textual. Los experimentos mostraron que esta diferencia afecta a la capacidad de completar el pipeline y, por ello, se registra junto al modelo evaluado.

El pipeline permite además configurar un modelo distinto para cada una de sus cuatro etapas (clasificación, intervención, orquestación y generación de respuesta), con reserva al modelo por defecto cuando no se especifica ninguno. Esto habilita, por ejemplo, un modelo más económico para las decisiones estructuradas y uno más capaz para el texto abierto que leerá el estudiante.

Si una salida no cumple el esquema, el modelo recibe el error de validación y puede reintentar. Si una etapa posterior falla, el resultado conserva las decisiones ya producidas; así el panel muestra dónde se detuvo el caso en vez de reducirlo a un error genérico.

\section{Decisiones de facilitación}

El pipeline operacionaliza la distinción, desarrollada en el estado del arte, entre analizar la discusión, decidir si conviene intervenir y escoger cómo hacerlo. Separar estas decisiones evita que la generación de un texto presuponga que intervenir era necesario. Las taxonomías siguientes convierten los conceptos revisados en valores que los nodos pueden devolver y que el panel puede mostrar.

La implementación usa cinco roles de facilitación independientes, cada uno con su propio agente y su propio grupo de técnicas (tabla~\ref{tab:mapeo-roles}). Los roles organizacional, intelectual y social proceden de la línea de investigación que se inicia con Paulsen \parencite{Paulsen1995} y aparece en varias taxonomías posteriores \parencite{Abdous2011,BlignautTrollip2003,DeWeverEtAl2010}; concentran, respectivamente, 5, 8 y 5 técnicas. A partir de esas categorías se añadieron dos extensiones de diseño. El rol afectivo reúne 5 técnicas de reconocimiento y refuerzo positivo, mientras que el moderador incorpora 7 técnicas de moderación pasiva, como señalar contenido inapropiado o con derechos de autor \parencite{KorreEtAl2025}. Esta separación mantiene las acciones de escalado fuera de los roles de facilitación activa.

El repertorio completo de técnicas se expone a todos los agentes de rol en un mismo orden, sin filtrar por rol ni por estado, lo que permite selecciones cruzadas entre roles cuando el contexto de un hilo lo justifica. Esta decisión asume el riesgo de sesgo de posición documentado en la literatura sobre listas largas presentadas a LLMs \parencite{ZhengEtAl2023}, riesgo que la evaluación declara como limitación abierta en lugar de mitigar sin datos empíricos previos.

El agente del rol moderador dispone de una acción adicional para solicitar la revisión del instructor. La llamada de marcado no se verificó contra un foro real, por lo que debe considerarse una ruta pendiente de validación y no una barrera de seguridad.

Para que la fase de clasificación pudiera decidir si intervenir hacía falta un vocabulario de estados que describiera las situaciones en que el sistema actúa. Se definieron seis estados mutuamente excluyentes (tabla~\ref{tab:estados}) \parencite{BlignautTrollip2003,AnEtAl2009}, cada uno asociado a un conjunto de roles y acciones posibles. Asignar una sola etiqueta por ejecución fue una decisión deliberada. Con una única etiqueta, cada estado lleva directamente a un conjunto de roles sin reglas adicionales de prioridad.

El estado \emph{estancada} requería una precisión. Un hilo puede estar en silencio porque los participantes están reflexionando, lo que no requiere intervención, o porque llegó a un punto de estancamiento real, lo que sí la requiere. Esa diferencia se resuelve leyendo si las últimas contribuciones avanzaban el debate o lo cerraban, y no con un umbral temporal. Los trabajos sobre tutoría y fracaso productivo indican que, en tareas con una solución identificable, intervenir antes del impasse puede interrumpir un proceso útil de exploración \parencite{VanLehn2011,Kapur2016}. El sistema traslada esa idea como hipótesis de diseño a discusiones abiertas, donde no siempre existe una respuesta correcta a la que converger; por eso el clasificador analiza la trayectoria del hilo y la validez de esta transferencia queda pendiente de evaluación.

El umbral de estancamiento es de 48 horas por defecto y se establece en la configuración. Los demás criterios, como participación dominada, trayectoria en declive y calidad del discurso, se describen de forma cualitativa en el prompt y dependen de la interpretación del modelo.

Cada uno de los cuatro tipos de salida del pipeline (clasificación, decisión de intervención, selección de rol y respuesta) incluye un campo de confianza auto-evaluada por el modelo, con valor por defecto 1.0. El campo se conserva como instrumentación experimental y no tiene efecto operativo. En los experimentos de este trabajo no se analizó su calibración frente a la exactitud observada, por lo que no se interpreta como una medida fiable de calidad.

\section{Prompts y extensibilidad}
\label{sec:prompts}

Para dar coherencia al comportamiento de los agentes entre sí, todos los prompts del sistema siguen una estructura de cuatro secciones. La sección \texttt{Persona} define la identidad del agente, mínima para los nodos internos del pipeline y más rica para los agentes de rol que producen texto que leerá el estudiante. La sección \texttt{Constraints} declara los límites de responsabilidad y las restricciones de comportamiento. La sección \texttt{Context} contiene la información variable por ejecución, como umbrales configurables, contenido del hilo y marcas de tiempo. La sección \texttt{Task} describe qué debe producir el agente, qué puede incluir en la salida y cómo llegar a ella \parencite{SikstromEtAl2022}.

El prompt del agente clasificador ilustra cómo se aplica esta estructura.

\begin{small}
\begin{verbatim}
# Persona
You are a learning scientist reading an asynchronous academic
discussion thread.

# Constraints
You do not decide whether to intervene - that belongs to the next
pipeline node. Accuracy here determines everything downstream.

# Context
Discussion context: {discussion_context}
Current timestamp: {current_timestamp}
Stalled threshold: {stalled_threshold_hours} hours without new posts

# Task
Read the thread and produce a structured classification across five
fields: state, trajectory, participation balance, discourse quality,
and inquiry phase.
\end{verbatim}
\end{small}

Durante una ejecución, las variables anteriores se completan y el contenido del hilo se envía como mensaje de entrada. Por ejemplo, el escenario \texttt{new} produjo el siguiente mensaje para el clasificador:

\begin{small}
\begin{verbatim}
Current timestamp: 2026-07-23T11:51:47+00:00
Topic: Privacy implications of large language models
Learning objectives:
- Identify privacy risks in LLM training data
- Evaluate current mitigation strategies
- Propose privacy-preserving alternatives

Posts:
- [2026-03-12T13:00:00+00:00] Prof. Garcia:
  This week we're discussing privacy in LLMs. What are the main
  risks when personal data appears in training sets?
\end{verbatim}
\end{small}

El modelo devolvió una instancia del esquema de clasificación con los siguientes valores:

\begin{itemize}
  \item Estado: \texttt{new}.
  \item Trayectoria: \texttt{never\_started}.
  \item Participación: \texttt{instructor\_centered}.
  \item Calidad del discurso: \texttt{formulaic}.
  \item Fase de indagación: \texttt{triggering}.
\end{itemize}

El registro de la ejecución conserva tanto el prompt del sistema como este mensaje, la salida estructurada y el razonamiento. De esta forma, el ejemplo no representa una instrucción aislada, muestra cómo la plantilla se combina con el contexto de un hilo concreto.

Los agentes de rol tienen una sección Persona más elaborada, con un perfil que define el tono y el registro de la respuesta; los nodos internos mantienen esa caracterización al mínimo.

El pipeline es un grafo dirigido donde cada nodo tiene un único punto de decisión. Un objeto de estado compartido fluye entre nodos acumulando la clasificación del hilo, la decisión de intervenir, el rol seleccionado y la respuesta generada.

Antes de entregar la respuesta, el agente de rol verifica que el texto no esté vacío, que indique la técnica aplicada y que no contenga ciertas expresiones de evaluación o calificación. Si alguna comprobación falla y quedan intentos, devuelve el control al orquestador con una descripción del problema.

El repertorio de técnicas se expone como una herramienta que los agentes de rol consultan durante la ejecución, en lugar de incluirlo completo en el prompt. Se construyó a partir del corpus revisado y está indexado por rol y estado; cada técnica declara su nombre, descripción, ejemplos y referencia. Los modelos sin soporte funcional de herramientas pueden completar las primeras decisiones, pero no la generación de respuesta en esta implementación.

El agente de rol dispone también de una herramienta de búsqueda web, guiada por el tema de la discusión, para encontrar ejemplos, definiciones o material de referencia pedagógicamente relevante. Es opcional dentro del razonamiento del agente, se invoca solo cuando encontrar contexto externo ayuda a elegir mejor la técnica o el tono de la respuesta.

El agente de rol reduce el volumen de contexto truncando el razonamiento heredado y devolviendo un ejemplo por técnica. Ante errores menores de formato, intenta reparar la salida estructurada antes de descartarla. Las herramientas externas devuelven un error descriptivo si su servicio no está disponible.

Cuando el agente de rol agota sus reintentos, el sistema conserva el historial parcial de mensajes de esa ejecución en lugar de descartarlo, lo que permite inspeccionar exactamente qué produjo el modelo en vez de solo un mensaje de error genérico. Un límite de una llamada por herramienta y por respuesta se impone en el propio código de la herramienta, no solo como instrucción en el prompt, lo que evita que un modelo que repite una llamada agote su presupuesto de reintentos en invocaciones redundantes.

\section{Fuentes de discusión y prueba con Open edX}
\label{sec:fuentes-discusion}

El objeto que procesa el sistema es una discusión académica, no una plataforma concreta. En los experimentos, la fuente principal son escenarios sintéticos y seis hilos históricos anonimizados cargados desde archivos versionados en el repositorio. Los hilos históricos se extrajeron de \textit{Dataset MOOC Forum edX} \parencite{AlarioHoyos2021Dataset}, que reúne foros de tres MOOC de programación y varias ediciones. La versión citada publica el archivo original, su licencia CC BY 4.0 y su suma de comprobación. El repositorio conserva el script de extracción, los 480 candidatos resultantes y los seis casos seleccionados. La DDA-0041 documenta la cadena completa y la selección manual.

El panel también puede solicitar hilos a un backend que implemente el contrato mínimo de entrada. Open edX se utilizó como prueba de ese límite porque organiza las discusiones como hilos y admite extensiones mediante plugins. Moodle, Canvas o Discourse requerirían su propio adaptador; no se implementaron en este trabajo.

El contrato declara solo lo que el pipeline necesita: texto inicial, comentarios con autor y marca temporal y algunos metadatos del hilo. Los objetivos de aprendizaje se entregan por separado, pues no todas las fuentes los proporcionan. El backend de Open edX adapta la respuesta de su API a ese contrato; los escenarios de evaluación construyen el mismo tipo de entrada sin pasar por un LMS.

El segundo problema fue el modelo de activación. Un webhook que Open edX llamaría al crear o actualizar un hilo habría requerido cambios de infraestructura (registro del webhook, reintentos, idempotencia) sin valor diferencial para la evaluación, y un bucle de polling habría introducido latencia artificial y un proceso en ejecución continua innecesario para un prototipo. Se optó por un modelo de activación bajo demanda donde un cliente externo activa el ciclo con el identificador del hilo que quiere facilitar, lo que hace el sistema compatible con una integración futura basada en eventos sin requerirla ahora.

El servicio expone una función interna y una capa REST que delega en ella (figura~\ref{fig:activacion-bajo-demanda}). Los scripts de evaluación y el panel usan esta lógica común. Se construyeron el backend y los puntos de comunicación necesarios para probar Open edX como fuente, pero no se completó un flujo extremo a extremo integrado en la interfaz habitual del docente. En particular, no se evaluaron la activación cotidiana desde el LMS, la revisión previa por el profesor ni la publicación segura de respuestas en un curso en producción.

\section{Entornos de ejecución}
\label{sec:entornos-ejecucion}

El servicio completo está desplegado tanto en Idril como en AWS EC2 (tabla~\ref{tab:environment-config}). En ambos entornos se ejecutan el mismo panel React, la API FastAPI y el pipeline descrito en este capítulo. Por tanto, se trata de mantener dos instalaciones equivalentes con configuraciones experimentales distintas, no de separar frontend y backend entre servidores.

La secuencia es la misma independientemente del entorno elegido (figura~\ref{fig:two-execution-environments}), y la separación permite iterar con modelos locales y externos de forma independiente. Una prueba prolongada o una configuración incorrecta en un entorno no bloquea el otro, y cada proceso mantiene su proveedor y catálogo de modelos. Los resultados, en cambio, sí se consolidan, ambos despliegues escriben los manifiestos experimentales en el mismo bucket y prefijo de S3. También comparten el proyecto de Logfire, el backend configurado para leer Open edX y un timeout de 1.800 segundos por ejecución del pipeline.

\subsection{Idril: modelos locales}

Idril es un servidor con Debian Linux, 28 CPU, 128 GB de memoria RAM y una GPU NVIDIA RTX 4000 con 20 GB de VRAM. La GPU permite servir mediante Ollama los modelos que se ejecutan en infraestructura de la universidad. El panel y la API se despliegan en el mismo servidor y son accesibles dentro de la red de la UCM.

Este entorno se reserva a perfiles de modelos locales. Su catálogo experimental incluye dos tamaños de Ministral 3 y dos de Qwen 3.5. La lista disponible y el modo de extracción estructurada se pueden ajustar sin afectar las pruebas con APIs externas. Las latencias no son directamente comparables con EC2: en Idril incluyen la inferencia sobre hardware local y en EC2 dependen de servicios remotos.

\subsection{AWS EC2: modelos externos}

La segunda instalación ejecuta el mismo panel y la misma API en una instancia EC2. En esta configuración, el pipeline envía las solicitudes a OpenRouter, que proporciona acceso a GPT-4o, Claude Haiku 4.5, GPT-4o-mini y DeepSeek V3.1 bajo una interfaz común. Sus registros permiten inspeccionar qué proveedor atendió una llamada, los tokens consumidos, la velocidad y el coste.

EC2 permite probar modelos externos sin alterar las variables ni los procesos activos en Idril. También ofrece un entorno accesible fuera de la red universitaria para mostrar el panel. El acceso a la aplicación está protegido mediante autenticación HTTP básica. Open edX está configurado como fuente en ambos despliegues, pero no determina su arquitectura ni forma parte de la comparación entre proveedores.

Los dos despliegues son, por tanto, bancos de prueba complementarios del mismo artefacto. No representan una división funcional del sistema ni completan el flujo de trabajo del docente dentro de un LMS. El capítulo siguiente analiza conjuntamente las ejecuciones conservando la procedencia de cada una.
