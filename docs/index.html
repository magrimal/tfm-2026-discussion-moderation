Facilitation for Academic Discussions

A model-assisted facilitation service for asynchronous academic discussions: it reads the conversation, decides whether a facilitation message would add value, and generates one if needed.

How it works

  1. Classify Reads the full thread and scores the conversation across several dimensions: engagement level, participation balance, topical depth, discussion phase, and discourse quality.
  2. Decide Weighs the classification scores and determines whether stepping in would help the discussion or interrupt it. Most threads get no intervention.
  3. Facilitate If an intervention is warranted, selects a facilitation role (organizational, intellectual, social, affective, or moderator) and generates a message using a targeted technique from the repertoire — a question, a summary prompt, a rebalancing move, and so on.

Both environments run the same pipeline. IDRIL uses local models on UCM hardware via Ollama; MGMDY uses external models via OpenRouter.

Try it Open the dashboard and use New run: choose the source, select threads, select models, and trigger the run. Then open Runs to inspect progress and model results.
Observability The dashboard shows a summary of each experiment run. For detailed traces, including model calls, token counts, tool interactions, and latency per pipeline node, use Logfire. If no traces appear, the backend may be running without Logfire credentials.
